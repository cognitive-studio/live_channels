# Meridian v3 architecture review

Andrew asked me to place the live-thread review into the durable channel so the crew can inspect, challenge, and ratify it.

## Decision required

**Affected scope:** Meridian’s root definition, Horizon sequence, source/branch doctrine, parity doctrine, agent authority, certification language, handoff protocol, Finance-domain structure, and canonical repository organization.

**Evidence and uncertainty:** The existing Meridian corpus contains a strong governance spine, but the FDH real-source work has changed the operating center. The hand-built harness proved useful for reconstruction and simulation, yet it also exposed that a non-canonical scaffold can lie about routing, flags, portals, data shapes, auth, and shared dependencies. The real FDH dev environment is the honest substrate. Some exact implementation details remain dependent on the current GHE repository state and engineering access.

**Consequence of proceeding without amendment:** Meridian risks institutionalizing a harness-first, surface-centric model after the program has already learned that the durable need is a whole-product, canonical-source, executable design-to-production relay. That would preserve separability at the seam and allow prose-only intent, parity drift, and ambiguous certification to re-enter the system.

**Authority needed:** Andrew should ratify the constitutional direction. Prudence should sequence the documentation and repository changes. Atlas-CS, Wibey, GenLD, Gem, and Lumi should pressure-test their respective architecture, implementation, design-system, strategic, and accessibility implications.

## Overall verdict

Meridian is conceptually strong, but its canonical documentation is now one generation behind what Meridian itself has learned.

The healthy spine is already present:

- product-level governance above the component system;
- context before redesign;
- parity separated from optimization;
- agents separated from instruments;
- provenance, uncertainty, and state completeness;
- UX treated as a governed product stage rather than a Figma deliverable.

The real-source FDH work changes the governing center:

> Meridian is no longer primarily a system for reconstructing a product into a separate parity harness. It is becoming a governed, whole-product design-to-production relay operating against canonical source.

This is not a conceptual restart. It is a constitutional amendment and a cull.

## 1. Correct the Horizon sequence

The current sequence appears to allow component-system resolution before the honest design-parity baseline. That is backwards. If components are replaced or resolved before the current product is frozen, then the baseline is already altered.

Recommended sequence:

```text
P0   Access and Environment Proof
 └─→ P1   Canonical Source Baseline
      └─→ P2   Whole-Product Parity Baseline
           ├─→ P3A  Product Context Reconstruction
           ├─→ P3B  Technical and Workflow Reconstruction
           └─→ P3C  Current-State Certification Assessment
                    └─→ P4   Component and Pattern Resolution
                         └─→ P5   Northstar Optimization
                              └─→ P6   Executable Handoff
                                   └─→ P7   Developer Verification
                                        └─→ P8   Release Evidence and Drift
                                             └─→ P9   Continuous Stewardship
```

Context and technical reconstruction may proceed in parallel after the honest environment exists. Neither should force the team back into a hand-built simulation.

## 2. Amend the canonical definition

Current phrasing around “assembled enterprise product surfaces” is too UI-centric for what Meridian now governs.

Recommended definition:

> **Meridian is the governed design-to-production relay, certification system, and stewardship layer for whole enterprise products. It operates against canonical source, preserves current-state parity, enables evidence-based optimization, produces executable engineering handoffs, and verifies that product intent survives implementation.**

## 3. Amend the source and branch doctrine

The old preferred model was effectively:

```text
production repository = read-only evidence
separate UX-stage repository = build location
mock data = standard operating substrate
```

That may remain an exception, but it should no longer be the preferred path.

Preferred model:

```text
canonical product repository
→ designated northstar/* UX branch
→ tagged handoff commit
→ engineering implementation branch
→ Engineering-owned QA and release
```

Authority boundary:

```text
UX:
- reads canonical source
- creates and pushes designated northstar/* branches
- cannot merge protected branches
- cannot deploy production

Engineering:
- reviews the handoff
- adapts and wires where necessary
- owns merge, QA, release, and production
```

A separate stage repository is permitted only when canonical branching is impossible, and must be labeled:

> **Non-canonical reconstruction workspace. Not valid evidence of source parity.**

## 4. Add Prerequisite 0 — Honest Environment

Access is part of product truth, not administrative trivia.

A Meridian run may not claim whole-product parity until it proves:

- canonical source commit;
- dependency installation;
- application startup;
- approved configuration;
- SSO/authentication;
- required API identities;
- role and entitlement resolution;
- representative non-production data;
- primary routes;
- real shared packages;
- known substitutions.

For FDH, record the now-held AD-group access as evidence, while keeping People API impersonation whitelist and RBAC consumer IDs as separate controls until proven.

Recommended artifacts:

```text
products/finance-data-hub/environment/
  ACCESS_RECEIPT.md
  LOCAL_ENVIRONMENT_RECEIPT.md
  KNOWN_ACCESS_GAPS.md
```

## 5. Replace the permissive fake-data rule with a two-tier doctrine

The earlier line “fake data is acceptable; fake behavior is not” is no longer sufficient. A harness can model plausible behavior while still lying about the real application.

Replace it with:

> **Synthetic data is appropriate for deterministic state coverage, edge cases, privacy protection, and unavailable-service simulation. Synthetic data does not satisfy the real-environment parity gate.**

Distinguish:

### Parity environment

- canonical source;
- real product architecture;
- real route and state structure;
- approved non-production services or contract-faithful substitutes;
- representative permissions;
- explicit deviation receipt.

### Simulation environment

- deterministic state forcing;
- loading, failure, empty, partial, and blocked scenarios;
- high-volume and long-label cases;
- inaccessible or destructive scenarios;
- component experimentation.

The hand-built harness remains valuable as a simulator, not the product room.

## 6. Promote the handoff artifact into an executable contract

The handoff must not remain prose-only. “Wiring notes in developer terms” are necessary but insufficient; prose-only intent reintroduces separability at the seam.

Recommended structure:

```text
handoff/
├── HANDOFF_CONTRACT.md
├── interaction-invariants.yaml
├── data-contract-map.yaml
├── state-matrix.yaml
├── impact-map.yaml
├── verification-plan.yaml
├── known-deviations.yaml
├── evidence/
└── HANDOFF_RECEIPT.yaml
```

The missing high-leverage artifact is the whole-product impact map:

```yaml
change:
  initiative: shop-redesign
  baseline_commit:
  handoff_commit:

directly_affected:
  - shop
  - global-search
  - access-request

indirectly_reviewed:
  - report-listing
  - data-listing
  - favorites
  - onboarding
  - responsive-shell

shared_components_changed:
  - component:
    consumers:

contracts_touched:
  - route
  - API
  - entitlement
  - telemetry

reviewed_no_change:
  - metrics
  - report-management
```

This turns the holistic requirement into a reviewable product delta.

## 7. Make the drift gate real without owning Engineering CI

A gate that depends on a pipeline we do not control is only a request. The first drift gate should therefore be a **release-evidence obligation**, not a presumed CI control.

Required statuses:

```text
UX VERIFIED
UX VERIFIED WITH DEVIATIONS
UX NOT VERIFIED
```

Required handoff receipt:

```yaml
handoff_tag:
dev_branch:
verified_commit:
environment:
verification_status:
invariants_checked:
deviations:
ux_reviewer:
engineering_reviewer:
product_reviewer:
verified_at:
```

Portable checks should run locally first:

```text
npm run verify:ux-handoff
```

or equivalent.

Maturation path:

```text
Manual evidence
→ portable verifier
→ optional CI check
→ required CI check
→ release-policy gate
```

We do not need to own Engineering’s pipeline to make drift visible, attributable, and non-silent. We need a baseline, executable invariants, required evidence, and named authority.

## 8. Create an immutable whole-product parity baseline before optimization

The first honest render of real FDH data should be tagged before any UX change.

Example:

```text
northstar/fdh-parity-baseline-001
```

Record:

- route coverage;
- working authentication;
- real data surfaces;
- known broken behavior;
- accessibility defects;
- state coverage;
- telemetry and timestamps;
- component versions;
- all substitutions.

Without this baseline, later defects cannot be reliably attributed as inherited, UX-introduced, developer-wiring, backend, or environment drift.

Shop may be the first meaningful change inside the branch. Shop must not define the boundary of the pilot.

## 9. Correct the authority doctrine

The phrase “agents decide; instruments execute” is useful as a distinction but too broad as governance.

Replace it with:

> **Agents deliberate, interpret, challenge, recommend, and may refuse. Instruments execute bounded operations. Decisions are made by the named human or delegated authority appropriate to the decision class.**

Suggested decision classes:

```text
D0 — reversible implementation detail
D1 — product interaction decision
D2 — architecture or shared-pattern decision
D3 — access, data, security, or compliance decision
D4 — production-release decision
```

Each class should name:

- authorized decider;
- required reviewers;
- veto holders;
- evidence threshold;
- receipt requirement.

No synthesizer or council should become sovereign by accident.

## 10. Qualify Meridian certification

Add a governing disclaimer:

> **Meridian certification is an evidence-backed product-readiness status under the Meridian protocol. It does not replace official Engineering, Accessibility, Security, Privacy, Legal, Compliance, or production-release approvals.**

Every certification must be scoped:

```yaml
certification:
  subject:
  product:
  routes:
  workflows:
  source_commit:
  environment:
  evidence:
  exclusions:
  exceptions:
  issued_by:
  issued_at:
  expires_or_review_due:
```

No product should be called “certified” without saying what, where, against which commit, under which exclusions, and for how long.

## 11. Separate Finance-domain coherence from product architecture

Data Hub and Finalytics remain separate Meridian instances.

Recommended structure:

```text
domains/
└── finance/
    ├── FINANCE_DOMAIN_CONTRACT.md
    ├── SHARED_PATTERN_LEDGER.md
    ├── TERMINOLOGY.md
    ├── TRUST_AND_PROVENANCE_POLICY.md
    └── CROSS_PRODUCT_DECISIONS.md

products/
├── finance-data-hub/
│   ├── PRODUCT_MANIFEST.yaml
│   ├── ENVIRONMENT_TOPOLOGY.md
│   ├── PARITY_BASELINE.md
│   └── handoffs/
└── finalytics/
    ├── PRODUCT_MANIFEST.yaml
    ├── ENVIRONMENT_TOPOLOGY.md
    ├── PARITY_BASELINE.md
    └── handoffs/
```

The Finance layer governs shared expectations. It must not become a shared runtime that erases product-specific architecture.

## 12. Preserve the foundational doctrine

Keep these principles:

- Context before redesign.
- Existing product as evidence, not unquestioned truth.
- Parity and optimization as separate states.
- The agent should not need to guess.
- No silent resolution of uncertainty.
- Production authority remains outside UX.
- Every transformation preserves provenance.
- Non-happy paths are part of product completeness.
- Product runs should improve later product runs.
- Sidebar/controller separated from runtime.

And:
